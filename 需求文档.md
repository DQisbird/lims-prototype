## 一、全局基础约定

### 1.3 角色体系（11可见 + 1隐蔽）

**管理层（4个）：** 实验室负责人、部门负责人、质量负责人、技术负责人

**关键岗位（5个）：** 授权签字人、内审员、设备管理员、配置管理员、质量监督员

**执行层（2个）：** 测试工程师、样品管理员

**隐蔽角色（1个）：** 隐蔽管理员（11个角色全部权限 + 时间调整特权，仅该账号登录后可见）

> 权限规则：一人多角色权限取并集；任务分配时自动检查人员是否具备对应领域检测授权。

### 1.4 编号规则（20种编号，全系统唯一）

**自动生成类（系统生成，校验唯一性）：**

| 编号 | 格式 | 生成时机 |
|------|------|---------|
| 任务编号 | `TZJC/JSJL-{领域}{YYYY-MM-DD}`（支持手动修改，可加后缀如-01避免同日同领域冲突，唯一） | 新建任务时生成 |
| 样品编号 | `{任务编号}-YP-{NN}` | 样品接收时 |
| 设备编号 | 系统自动生成 | 新增设备时 |
| 记录编号 | `{任务编号}.{表单缩写} V{版本号}` | 上传文件后 |
| 报告编号 | `{任务编号}.JCBG V{版本号}` | 上传报告后 |
| 技术记录编号 | `{任务编号}.{表单缩写} V{版本号}` | 上传文件后 |
| 变更编号 | `{原报告编号} V{版本号+1}` | 变更审批通过后 |

**手写类（支持手填和修改，保存时校验唯一性）：**

| 编号 | 格式建议 |
|------|---------|
| 评审编号 | 手动填写，唯一；格式建议：`TZJC/JSJL-{评审类型}{YYYY-MM-DD} V{版本号}`，后续可关联任务编号 |
| 合同编号 | `TZJC-JCHT-{YYYY-MM-DD}` |
| 委托编号 | `{任务编号}.WTSQ V{版本号}` |
| 受控号 | `TZJC/{类型}+序号+版本` |
| 方法编号、偏离编号、不符合编号、投诉编号、计划编号、供应商编号、调查编号、工单编号、标准编号 | 手写 |

> **编号时序链**：委托创建 → 编号暂空 → 审批通过 → 通知创建人新建任务 → 手动创建任务（选择关联委托，自动带入委托信息）→ 生成任务编号（支持手动修改）→ 委托编号自动回填 → 后续编号依赖任务编号生成。

### 1.5 日期格式

全系统统一 `YYYY-MM-DD`，编号中亦如此，禁止 `YYYYMMDD` 等其他格式。

## 2.5 待办工单

> 跨模块统一入口，展示所有分配给当前用户的待处理工单，按类型分组。

### 页面：待办工单列表

**筛选条件：** 工单类型、客户名称（模糊）、分配日期范围

**列表展示：** 按工单类型分组折叠，每组表头显示：类型名称 + 待处理数量。

**工单类型与来源：**

| 工单类型 | 来源模块 | 触发时机 | 处理入口 |
|---------|---------|---------|---------|

| 记录评审 | 2.3.8 测试记录评审 | 上传测试记录后 | 点击跳转任务详情→记录评审环节 |
| 问题报告评审 | 2.3.10 回归测试问题报告评审 | 上传回归测试记录后 | 点击跳转任务详情→问题报告评审环节 |
| 报告评审 | 2.3.12 检测报告评审 | 上传检测报告后 | 点击跳转任务详情→报告评审环节 |

| 字段 | 类型 | 说明 |

## 三、核心业务流程（第2章检测任务管理 — 最重要的章节）

### 3.1 端到端流程总览

```
委托审批通过
    ↓
┌─────────────────────────────────────────────────────────┐
│ 第2章 检测任务管理 — 阶段流水线                           │
│                                                         │
│  A.前期准备 → B.方案 → C.实施 → D.问题处理 → E.报告 → F.结项 │
│                                                         │
│  全程伴随：                                              │
│  ① 评审通过→文件自动定稿纳入配置管理                      │
│  ② 每次阶段流转→自动给质量监督员发监督工单（不阻塞）       │
└─────────────────────────────────────────────────────────┘
    ↓
任务完成
```

### 3.2 六阶段推进规则

| 阶段 | 名称 | 包含环节（按顺序） | 推进条件（全部满足才能进入下一阶段） |
|------|------|-------------------|-------------------------------------|
| **A** | 前期准备 | 样品接收 → 任务派工 | ①样品已接收 ②派工确认 |
| **B** | 方案 | 测试方案 → 方案评审 → 用户确认方案 → 测试配置及环境确认 | ①方案评审全员通过 ②用户确认文件已上传 ③配置/环境确认表已上传 |
| **C** | 实施 | 测试记录（含问题报告）→ 测试记录评审 | 记录评审全员通过；无问题→进E报告；有问题→进D问题处理 |
| **D** | 问题处理 | 回归测试记录 → 回归测试问题报告评审 → **循环** | 问题修复=是 且 新问题=否 → 进E报告；否则继续循环 |
| **E** | 报告 | 检测报告 → 检测报告评审 → 用印审批 → 出具报告 | ①报告评审全员通过 ②用印审批四人全部通过 ③最终版报告上传 |
| **F** | 结项 | 归档 → 客户满意度调查 | ①确认归档 ②满意度调查（可跳过）→ 任务完成 |

### 3.3 阶段流转详细逻辑

#### A. 前期准备阶段

```
[2.3.1 样品接收]
  操作：确认样品已接收 → 关联3.1样品管理模块
  流转：确认 → 推进到"任务派工"
        ↓
[2.3.2 任务派工]
  弹窗：主检人员(下拉,仅该领域授权人员) + 复核人员(下拉) + 计划开始日期 + 计划完成日期
  校验：主检人必须具备该领域授权；主检人和复核人不能同一人；监督期内人员需指定监督人
  流转：确认派工 → 推进到"测试方案"
```

#### B. 方案阶段

```
[2.3.3 测试方案]
  弹窗：方案文件(文件上传) + 方案说明(多行文本)
  流转：上传 → 自动生成编号 {任务编号}.CSFA V{版本号} → 推进到"方案评审"
        ↓

### 3.4 任务生成规则

**触发条件：** 第1章委托审批通过 → 系统通知委托创建人，由创建人手动新建任务

**生成规则：**
- 新建任务时选择关联已通过的委托，委托信息（委托方、检测对象、检测领域等）自动带入并只读展示
- 用户自由决定多个检测领域时按领域拆分多个任务或合并为一个任务
- 委托审批通过后仅通知委托创建人，不自动生成任务；任务由人工创建
- 任务初始阶段 = "样品接收"（阶段A的第一个环节）
- 任务编号自动生成：`TZJC/JSJL-{领域}{YYYY-MM-DD}`（支持手动修改，可加后缀如-01避免同日同领域冲突，唯一）

**编号生成时序：**
```
委托创建 → 委托编号暂空（或手写）
    ↓
委托审批通过 → 通知创建人 → 手动新建任务（选择关联委托）
    ↓
新建任务 → 任务编号生成（支持手动修改）
    ↓
任务编号生成后 → 委托编号自动回填（{任务编号}.WTSQ V{版本号}）
    ↓
后续编号（样品编号、记录编号等）均依赖任务编号生成
```
